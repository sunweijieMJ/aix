## 职责

负责 API 接口开发指导，包括 RESTful 设计规范、版本管理、请求日志和限流策略。

---

## RESTful 设计原则

### URL 规范

- 使用 **kebab-case** 命名
- 资源名使用 **复数**
- 完整 CRUD 动词对应：GET（查询）、POST（创建）、PUT（更新）、DELETE（删除）
- 嵌套资源最多 **2 层**: `/users/:id/orders`
- 具体路由实现由框架层预设定义

### 响应格式原则

- 所有 API 响应使用**统一的 JSON 结构**（包含状态码、消息、数据三要素）
- 分页响应需额外携带 `pagination` 元信息（page / pageSize / total）
- 具体响应格式由框架层预设定义，base 层不规定实现细节

### HTTP 状态码

| 状态码 | 含义 | 使用场景 |
|--------|------|---------|
| 200 | 成功 | GET / PUT / DELETE 成功 |
| 201 | 创建成功 | POST 创建资源成功 |
| 204 | 无内容 | DELETE 成功且无需返回数据 |
| 400 | 请求参数错误 | 参数校验失败 |
| 401 | 未认证 | 缺少或无效的认证信息 |
| 403 | 无权限 | 认证通过但无访问权限 |
| 404 | 资源不存在 | 请求的资源未找到 |
| 409 | 冲突 | 资源状态冲突（如重复创建） |
| 422 | 不可处理 | 参数格式正确但语义错误 |
| 429 | 请求过多 | 触发限流 |
| 500 | 服务器内部错误 | 未预期的服务端异常 |

## API 版本管理

### 版本策略

| 方式 | 格式 | 适用场景 |
|------|------|---------|
| **URL 前缀**（推荐） | `/api/v1/users` | 简单直观，适合大部分项目 |
| Header 版本 | `Accept: application/vnd.api.v1+json` | 精细控制，适合开放平台 |
| 查询参数 | `/api/users?version=1` | 不推荐，缓存不友好 |

### 版本升级原则

- 新增字段、新增接口 → 不升级版本（向后兼容）
- 移除字段、修改字段类型、修改行为 → 升级 MAJOR 版本
- 旧版本 API 至少维护 **6 个月**过渡期
- 过渡期内旧版本返回 `Deprecation` 响应头

```
Deprecation: true
Sunset: Sat, 01 Jan 2027 00:00:00 GMT
Link: </api/v2/users>; rel="successor-version"
```

## 请求日志规范

### 必须记录的信息

| 字段 | 说明 | 示例 |
|------|------|------|
| `requestId` | 请求唯一标识（用于链路追踪） | `req-a1b2c3d4` |
| `method` | HTTP 方法 | `POST` |
| `path` | 请求路径 | `/api/v1/users` |
| `statusCode` | 响应状态码 | `200` |
| `duration` | 请求耗时 (ms) | `42` |
| `userId` | 当前用户（如有） | `user-123` |
| `ip` | 客户端 IP | `192.168.1.1` |

### 日志级别

```
200-299 → info（正常请求）
400-499 → warn（客户端错误）
500-599 → error（服务端错误）
```

### 脱敏规则

- **禁止**记录：密码、Token、Cookie、信用卡号
- **脱敏**记录：手机号（`138****1234`）、邮箱（`z***@example.com`）
- 请求体超过 **1KB** 时截断，只记录关键字段

### Request ID 传递

```
客户端 → X-Request-Id: req-xxx → 网关 → 服务A → 服务B
                                          ↓
                                  日志中统一记录 requestId
```

- 客户端未传时，网关/服务端自动生成
- 所有下游调用透传 `X-Request-Id`
- 错误响应中返回 `requestId`，便于排查

## 限流规范

### 限流策略

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| **固定窗口** | 每 N 秒允许 M 次请求 | 简单场景 |
| **滑动窗口** | 过去 N 秒内不超过 M 次 | 精确控制 |
| **令牌桶** | 恒定速率填充，允许突发 | 通用推荐 |

### 限流维度

| 维度 | 说明 | 典型限制 |
|------|------|---------|
| IP | 防止单 IP 暴力请求 | 100 次/分钟 |
| 用户 | 防止单用户滥用 | 60 次/分钟 |
| 接口 | 保护高成本接口 | 10 次/分钟（如发送验证码） |
| 全局 | 保护系统整体 | 10000 次/分钟 |

### 限流响应

```
HTTP/1.1 429 Too Many Requests
Retry-After: 30
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1711814400

{
  "code": "RATE_LIMITED",
  "message": "请求过于频繁，请 30 秒后重试"
}
```

### 需要限流的高风险接口

- 登录/注册/发送验证码
- 文件上传
- 数据导出
- 第三方 API 调用（如短信、邮件）
- 批量操作接口
